# 食品饮料电商产品内容策略

## 工作原则

- 以抖音电商为首要场景，同时保留可迁移到其他电商平台的结论。
- 区分产品事实、外部事实、用户反馈和策略推断；不得把推断写成事实。
- 使用用户提供的链接、文件和资料，并用可用的网络或浏览工具补充公开信息。
- 不虚构销量、排名、认证、评论、成本、趋势或来源。
- 关键数据缺失时只完成能够得到证据支持的模块，并在“数据缺口”中说明；不要自行填入行业均值。
- 默认输出中文Markdown。除非用户另有要求，事实来源统一汇总在文末，不在正文逐句堆叠链接。
- 对食品功效、营养、成分和比较型宣传执行严格合规检查。

## 执行流程

### 1. 建立项目档案

整理产品名称、子品类、阶段、配料、口味、规格、包装、产地、工艺、保质期、保存方式、食用场景、目标人群、品牌定位、渠道、售价、成本、库存和毛利目标。

列出已收到与缺失的资料。只追问会阻止用户核心目标的少量关键问题；否则继续完成有数据支持的部分。

### 2. 提炼卖点体系

从产品事实出发，依次形成：

1. 产品事实与证明材料
2. 功能利益与感官利益
3. 情绪利益与场景利益
4. 目标人群及购买触发点
5. 相对竞品的差异化
6. 一句话定位、核心卖点、辅助卖点和传播主张

每个核心卖点必须绑定事实或证据。将无法证明的表达降级为待验证假设。

### 3. 调研市场、竞品、评论与热点

- 默认选择5–8个有效竞品；样本不足时使用实际数量并说明。
- 比较定位、人群、价格带、规格、配料、口味、工艺、主推SKU、内容钩子、促销机制、好评原因和差评机会。
- 对抖音评论进行去重、主题归类和语义复核；保留样本量及匿名化典型原话。
- 使用近30天观察快速热点，近90天观察稳定趋势；记录实际调研日期。
- 只采用与产品、人群和购买场景自然关联的热点。
- 记录来源标题、平台、URL或文件名、调研日期及可获得的样本量。

评论文件较大时先运行：

```bash
python scripts/analyze_comments.py INPUT --output comments-analysis.md
```

将脚本统计作为初筛，再人工复核上下文、反讽、否定和少数意见；不要把关键词命中直接当作消费者结论。

### 4. 设计SKU并测算利润

设计引流款、尝鲜装、主推款、利润款、混合口味装、囤货装、场景套装和加购/赠品款。并非每个项目都需要全部角色。

仅在相关输入存在时计算：

- 商品成交价与折扣
- 产品、包装、履约、运费、赠品及其他变动成本
- 平台佣金
- 达人佣金
- 服务商佣金比例及每单固定服务费
- 单笔贡献利润、贡献利润率和盈亏平衡获客成本

批量测算时运行：

```bash
python scripts/calculate_sku_profit.py sku-input.csv --output sku-profit.md
```

明确佣金的计提基数。默认按实际成交价计提；若合同口径不同，使用用户提供的口径并写入报告。缺少关键字段时不输出伪精确利润。

### 5. 形成内容策略与完整脚本

建立覆盖痛点解决、产品证明、食用场景、感官体验、配料工艺、用户口碑、对比测评、热点借势、情绪共鸣和促销转化的内容矩阵。

让每条重点脚本对应具体人群、SKU、卖点和证据。每条完整脚本至少包含：目标、建议时长、前3秒钩子、逐段口播、分镜/景别、人物或产品动作、字幕、声音建议、证据展示、CTA及合规提示。

### 6. 输出与自检

按报告模板生成单一Markdown文档。交付前检查：

- 卖点是否均有事实或来源支撑
- 用户洞察是否注明样本边界
- SKU角色、价格和内容是否互相对应
- 利润是否计入平台、达人和服务商佣金
- 脚本是否具备可拍摄的口播与分镜
- 食品宣称是否经过合规检查
- 数据缺口、风险和来源是否完整列出

## 输入兼容

接受文本说明、Markdown、网页链接、商品/视频链接、图片、PDF、CSV、TSV、XLSX、JSON和评论复制文本。使用适合文件类型的可用工具提取内容；保留原始文件名和字段含义。

若用户只要求某一模块，例如评论洞察或SKU测算，则只输出该模块及必要的依据、风险和来源，不强制生成完整报告。
